# hub NEEDS —— 定盘星

> 这个项目从 MVP spec（2026-07-09）中途硬转向（2026-07-12 推翻落地层），A 阶段在新架构下
> 建起来了，但"整个项目要什么、C 该长啥样"一直没写下来。这份文件补上那颗定盘星，
> 让后面的阶段（尤其 C）别再在机制上打转。用户明确取消/替换需求后允许覆盖更新。

## 真需求锚

我一个人、多台机器、同时用 **Claude Code 和 Codex**（以后可能加 opencode）；想从
"记忆 / skill / 插件 / 习惯散在各工具各机器、换机就丢、改一处别处不同步"，变成
"有一个**不会丢的单一金库**，能把家当收进去、也能装回**任何机器任何工具**，
**本地一处改动到处生效**"——为了不再手动搬运、不再换机重建、不再各复制体分岔。

**首要目的是方便本地发现 + 分发（我自己的几个工具）；别人从插件市场方便装是次级需求。**

## 三个阶段的位置

| 阶段 | 是什么 | 状态 |
|---|---|---|
| **A 提取器**（`hub/`） | 把本机家当收进金库（记忆 / 自写 skill / 自写插件仓快照 / agents 配置） | **已做完，合进 main**，218 测试，真金库已装 49 记忆 + 30 skill + 6 插件快照 |
| **C 加载器 skill** | 把金库装回工具、本地一处改到处新、跨设备人工闸门、符号根展开 | **没开始**——这是现在最大的开口，也是"本地发现+分发"这个首要需求的落点 |
| **B 加密层** | 挡着 `chats/` 和 sensitive 记忆 | **没开始** |

## 需求四组（回头版，带"做了没"）

### ① 第一版非做不可（核心闭环）

- 家当收进一个**不会丢**的金库 —— **A 已做**
- 我一处放的东西，我自己的 Claude / Codex 都能**发现并用**；本地改一处，各工具下次都新
  —— **C，没开始**（首要需求就是这条）
- 在任一工具里改了，改动能**回金库、不分岔** —— 镜像语义 + `shared/` 优先**已设计进 A**；
  但"共享区回吸 / 提升"是 **C，没开始**
- 换机能照金库**还原** —— 备份区 A 已备，但"装回来"是 **C，没开始**

### ② 必须达到的及格线

- **绝不泄密**：密钥路径永不读、`sensitive` 记忆不入库 —— **A 已硬做**（guard + 单 Writer + 先验后写）
- **绝不因配置错误把金库清空** —— **A 已硬做**（collect 先验后写，那个 CRITICAL 已堵）
- 跨设备**零冲突** —— A 的"每台机只写自己目录"已保证（但只有一台机，**没真验过多机 sync**）
- 装回工具时**符号根能正确展开**成本机路径 —— **C，没开始**

### ③ 做法可以替换（下一步 brainstorming 再挑，别现在拍）

C 用什么机制把东西装回工具，候选：
- **hub 自己的加载器 skill**（现有设计，SCHEMA.md 已把契约写死）
- **符号链接指过去**（本地单源，最轻）
- **npx skills**（跨 agent 包管理，默认 symlink 单源）
- **superpowers 壳**（每平台原生安装机制，主要给"发别人"）
- **rulesync**（当中间产物）

> **rulesync 的定位（2026-07-16 翻文档核实过）**：它是 **C 的候选，不是 A 的替代品**。
>
> **A（提取/备份）方向堵死**——不是性价比低，是没这功能：
> - **plugin：rulesync 没有"插件包"概念**（无 marketplace / plugin.json），提取 plugin 干不了。
> - **skill：`import` 不覆盖 skill**。它 `generate`（往外吐）是全八项，但 `import`（往里收）
>   有意做窄，官方例子/源码只有 rules/permissions；DeepWiki 明说 import 只做 rules/permissions，
>   generate 才是全八项。没证据能 import skill。
> - 加上它不建模记忆、不快照插件仓源码、没有 hub 的密钥硬闸。→ **A 自研，定死。**
>
> **C（分发到工具）方向它不错**：
> - skill 是**目录原样保留**（`.rulesync/skills/*/SKILL.md` + `references/`/`scripts/` 都留，
>   核实推翻了"可能有损重生成"的旧担心），分发不会毁 skill；generate 覆盖十几个工具，加新平台近乎白送。
> - 短板两条：**记忆它不建模**（仍需自写加载器装记忆）、**平台专属 hook 会丢**
>   （spike 实测 opencode 掉 preCompact、Codex config 空）。
> - **倾向**：主闭环（本地）用不用它待 brainstorming 比；它最稳的价值在次级需求
>   （分发给别人 / 一次铺很多工具 / 加新平台）。

### ④ 这一版先不做 / 次级

- **别人从插件市场方便装** —— 次级需求，排在本地闭环之后
- **加密层 B** —— `chats/` 和 sensitive 记忆被它挡着
- **平台专属 hook 跨平台分发**（compact-plus 那种 PreCompact/SessionStart）—— 平台能力墙，
  opencode/Codex 没这事件，**做不了**
- 多机 NAS sync 真验、`scope` 打标（49 条全 `global`）、Codex 自有插件收集 —— 都靠后
